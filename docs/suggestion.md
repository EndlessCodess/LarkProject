# 项目建议：优先选择方向 C，并吸收方向 B 的一部分能力

## 一句话建议
如果你希望项目既有“极客感”，又能在比赛里体现出明显差异化，我建议你以 **方向 C：嵌入式 / CLI 实时卡片式知识提示** 为主线，同时适度吸收 **方向 B：会议场景中的知识与行动项提取能力**，形成一个更完整、也更像真正 Agent 的作品。

可以把项目定位成：

**一个驻留在开发者工作流里的企业知识 Agent，在用户执行命令、查看报错、切换任务或进入特定上下文时，主动推送相关的飞书知识卡片、历史解决方案、API 文档、会议结论与行动项。**

---

## 为什么我更建议你做这个方向

### 1. 方向 C 最符合“极客气质”
你提到你个人是“以极客为目标的人”，那方向 C 天然更贴合你的风格，因为它：
- 更偏工程系统与开发效率工具，而不是纯产品展示。
- 更容易体现“上下文感知”“命令行工作流增强”“实时触发”“卡片化知识分发”等有技术味道的能力。
- 在评委视角里，更容易显得不是一个普通的聊天机器人套壳，而是一个真正嵌入工作流的 Agent。

### 2. 方向 C 差异化更强
A、B、D 都比较容易被做成“文档总结 + LLM 提炼 + 卡片输出”的常见方案。C 的优势在于：
- 触发时机更自然：不是用户主动问，而是系统在合适时机主动提醒。
- 场景更具体：开发者遇到报错、执行命令、切换代码模块时，最需要即时知识支持。
- 展示更有记忆点：CLI 中触发、飞书卡片联动、知识引用追溯，这类演示很容易留下印象。

### 3. 方向 C 更容易做出“知识分发”而不是“知识问答”
赛题反复强调，重点不只是 Q&A，而是“整合与分发”。

方向 C 很适合回答这个问题：
- 为什么用户不搜？因为他当下正在终端里忙，不会中断流程去查资料。
- 为什么普通问答不够？因为用户未必知道该问什么。
- Agent 的价值是什么？在正确时间，把正确知识，以最短形式送达。

这正是赛题最想看到的东西。

### 4. 技术闭环比较完整，也适合个人完成
个人参赛时，最怕项目面太大。C 方向可大可小：
- 小可做成“命令/报错 -> 召回知识 -> 输出提示卡片”的最小闭环。
- 大可扩展到“跨飞书文档、历史讨论、会议纪要、任务进展”的多源知识整合。

这意味着你可以先完成核心，再逐步加分，而不是一开始就陷入过度设计。

## 推荐的项目定义

### 核心定位
一个面向知识工作者，尤其是开发者/技术团队的上下文感知型 Agent。

### 核心目标
把分散在飞书中的文档、会议纪要、群聊结论、历史问题处理记录，转化为可被实时召回的结构化知识，并在 CLI 或协作节点中主动分发。

### 典型使用场景
#### 场景 1：终端报错时主动给解决卡片
用户执行命令后出现错误日志，Agent 自动：
- 识别错误类型
- 召回相关 API 文档 / 团队内部踩坑记录 / 历史群聊结论
- 输出简明知识卡片
- 附带来源链接

#### 场景 2：进入某项目模块时推送上下文
用户进入特定目录、切换分支、运行某服务时，Agent 自动给出：
- 该模块的设计文档
- 近期变更说明
- 相关负责人
- 常见问题
- 历史会议中的注意事项

#### 场景 3：会议后行动项同步到开发工作流
这是吸收方向 B 的部分。
会议结束后，Agent 自动从纪要中提取：
- 决策结论
- Action Items
- 对应负责人
- 截止时间
- 关联知识链接

然后在开发者下一次进入相关项目上下文时，主动提示这些事项，而不是让会议信息沉没。

## 为什么这条路线更容易拿到“产品价值 + 极客实现”的平衡

### 产品价值层面
它解决的是一个非常真实的问题：
- 企业知识很多，但不在用户真正需要时出现。
- 人们不是缺知识，而是缺“低摩擦获取知识”的方式。
- 传统搜索依赖用户主动输入关键词，而用户经常不知道该搜什么。

### 极客实现层面
这个项目可以体现很多很“硬”的设计点：
- 上下文感知触发
- 多源知识抽取与结构化
- 规则 + LLM 混合路由
- 检索增强生成
- 卡片化输出
- 可追溯引用
- CLI 与飞书协作联动

它会让你的作品看起来像一个“真正被工程师使用的系统”，不是只会聊天的 demo。

## 推荐的 MVP 范围
我建议你不要一开始就追求全能，而是做一个非常清晰的最小可行版本。

### MVP 核心闭环
1. 从飞书侧准备一批知识源：
   - 产品/项目文档
   - 历史 FAQ
   - 几条群聊总结
   - 一份会议纪要
2. 在 CLI 里监听两类触发：
   - 命令执行
   - 错误输出
3. 做一个知识提取层，把原始内容转成结构化条目：
   - 问题
   - 解决方案
   - 适用模块
   - 来源链接
   - 更新时间
4. 做召回与排序：
   - 根据报错关键词、目录上下文、当前任务进行匹配
5. 输出结果：
   - 本地终端简要提示
   - 飞书消息卡片 / 结构化展示
6. 保留引用来源，证明不是“编出来的答案”。

### MVP 最好只保证 3 个高质量 demo
- 一个报错修复 demo
- 一个目录上下文提示 demo
- 一个会议行动项联动 demo

只要这 3 个 demo 做得扎实，已经足够构成一个完整作品。

## 建议你如何定义“知识提取”
这个题里，“知识提取”不能只理解成摘要。

我建议你把它定义为：

**从文档、会议、聊天与任务等原始协作信息中，识别出可复用、可追溯、可分发、可触发的知识单元，并将其结构化为适合在具体工作场景中被主动使用的内容。**

这里的“知识单元”可以包括：
- 问题 - 解决方案对
- 决策结论
- Action Item
- 风险提醒
- 模块使用指南
- 历史故障经验
- API 注意事项

这个定义会比简单“摘要/问答”高级很多，也更贴近赛题要求。

---

## 你可以强调的创新点
如果你做这个项目，建议重点强调以下创新点：

### 1. 从“被动搜索”变成“主动分发”
不是用户问了才答，而是系统感知上下文后主动给知识。

### 2. 从“文本问答”变成“工作流嵌入”
知识不是停留在聊天框，而是进入终端、会议、任务流。

### 3. 从“文档检索”变成“结构化知识单元触发”
不是简单检索段落，而是提取出可执行、可落地的知识对象。

### 4. 从“答案生成”变成“带来源的可信辅助”
每个提示都带上原始文档/纪要/讨论来源，增强可信度。

## 评委可能会关心的问题，你可以提前准备答案

### 问题 1：为什么不用普通搜索？
你的回答可以是：
普通搜索需要用户知道搜什么，但真实工作中，用户常常不知道问题出在哪，也没空切换界面去搜索。这个 Agent 的价值在于利用上下文主动提示，降低知识获取摩擦。

### 问题 2：为什么不是一个普通聊天机器人？
你的回答可以是：
因为这个系统的核心不是对话，而是“在合适时机分发合适知识”，输出形态也不是只有聊天，而是 CLI 提示、飞书卡片、行动项联动等嵌入式形式。

### 问题 3：怎么证明有效？
你的回答可以是：
可以通过任务完成时间、错误定位时间、知识命中率、行动项遗漏率下降等指标来验证，并与传统搜索/人工翻文档方式做对比。

---

## 你应该避免的坑

### 1. 不要把项目做成泛泛的智能问答
如果最后只是“接一个知识库聊天机器人”，会很难突出赛题特色。

### 2. 不要想一口气覆盖所有办公场景
个人项目最重要的是“聚焦 + 完整闭环 + 展示亮点”。

### 3. 不要只讲模型能力，不讲业务价值
一定要反复强调：
- 节省时间
- 降低切换成本
- 减少知识沉没
- 减少行动项遗漏

### 4. 不要忽略效果验证
这个比赛不是只看功能能不能跑，还看你有没有证明它真的有价值。

---

## 如果你想更稳一点的备选方案
如果你后面发现 CLI 集成实现成本过高，或者飞书/终端联动不够稳定，那么备选可以转向：

### 备选：B 方向“会议场景全链路助手”
原因是：
- 数据结构天然清晰：会前、会中、会后。
- 业务故事非常完整。
- 容易展示 Action Item 自动提取、知识关联、任务分发。
- 评委也更容易理解。

但从“极客感”和差异化来看，我仍然建议你主打 C，B 作为补充能力模块。

---

## 最终建议结论

### 我的综合建议
**主选 C，辅以 B。**

### 具体打法
- 主线：做一个 `CLI / 上下文感知 / 主动分发` 的知识 Agent。
- 补强：加入 `会议纪要 -> Action Item -> 开发上下文提醒` 这条链路。
- 展示重点：不是“我能回答问题”，而是“我能在你需要之前，把知识送到你手边”。

### 为什么适合你
如果你希望作品既体现工程能力，又体现产品思考，还能保留明显的极客风格，这条路线是最合适的。

它会让你的项目同时具备：
- 有明确场景
- 有技术门槛
- 有产品差异化
- 有比赛叙事
- 有可验证价值
