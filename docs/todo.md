# 飞书 OpenClaw 赛道前置工作 Todo

## 一、先把题目理解清楚
- [ ] 通读赛题文档，明确核心目标不是“做一个问答机器人”，而是“做一个企业办公知识整合与分发 Agent”。
- [ ] 提炼赛题三大挑战：
  - [ ] 定义“知识提取”是什么。
  - [ ] 构建知识应用方式，而不只是对话问答。
  - [ ] 证明效果和业务价值。
- [ ] 明确交付物至少包含：
  - [ ] 产品/方案说明
  - [ ] 可运行 Demo
  - [ ] 效果验证方案

## 二、先选方向，不要一上来就写代码
文档给了 4 个参考方向：
- [ ] 方向 A：团队周报总结与洞察
- [ ] 方向 B：会议场景全链路助手
- [ ] 方向 C：CLI 实时卡片式知识提示
- [ ] 方向 D：团队事项推进追踪自动化

建议先完成以下决策：
- [ ] 评估自己更擅长哪类能力：产品设计、工程落地、Agent 编排、知识处理、数据验证。
- [ ] 判断哪个方向最适合个人参赛、最容易在有限时间内做出亮点。
- [ ] 明确最终只主攻 1 个主方向，其他方向最多作为扩展能力点展示。

## 三、定义你的“知识”范围
这是这道题最关键的前置工作。
- [ ] 先写清楚你的系统要处理哪些知识源：
  - [ ] 飞书 Docs / 文档
  - [ ] 飞书 Minutes / 会议纪要
  - [ ] 群聊消息 / 评论
  - [ ] 任务、待办、项目记录
  - [ ] 本地 CLI 上下文或开发行为（如果走 C 方向）
- [ ] 设计“原始信息 -> 可用知识”的转换链路：
  - [ ] 采集
  - [ ] 清洗
  - [ ] 结构化
  - [ ] 标签化
  - [ ] 检索 / 召回
  - [ ] 分发 / 触发
- [ ] 明确知识最小单元是什么：
  - [ ] 摘要
  - [ ] Action Item
  - [ ] 结论
  - [ ] 风险点
  - [ ] 决策记录
  - [ ] FAQ / 指南卡片
- [ ] 明确哪些信息必须保留原文链接，方便追溯与验证。

## 四、先设计用户价值，不要只设计模型流程
- [ ] 写出目标用户是谁：开发者、项目经理、团队负责人、会议参与人、普通员工。
- [ ] 写出用户的高频痛点：
  - [ ] 信息分散
  - [ ] 不知道去哪找
  - [ ] 不知道该搜什么关键词
  - [ ] 会后行动项容易遗漏
  - [ ] 终端工作流中断，知识无法及时触达
- [ ] 明确你的 Agent 帮用户节省的到底是什么：
  - [ ] 查找时间
  - [ ] 沟通成本
  - [ ] 上下文切换成本
  - [ ] 事项遗漏率
  - [ ] 知识沉没损耗
- [ ] 用一句话写出项目定位，例如：
  - [ ] “把分散在飞书里的办公知识，在对的时间，以对的形式，分发给对的人。”

## 五、先做信息架构与 Agent 流程图
- [ ] 画出系统最小闭环：
  - [ ] 输入源是什么
  - [ ] Agent 如何提取知识
  - [ ] 如何判断什么时候触发
  - [ ] 输出给谁
  - [ ] 输出形式是什么
- [ ] 确定输出形态，尽量不要只有聊天窗口：
  - [ ] 飞书消息卡片
  - [ ] 会后摘要
  - [ ] 周报卡片
  - [ ] CLI 终端提示
  - [ ] 自动待办分发
- [ ] 设计 2~3 个高价值典型场景作为 Demo 主线。

## 六、技术前置准备
- [ ] 熟悉 OpenClaw 的能力边界、接入方式和适合的 Agent 形态。
- [ ] 确认是否需要：
  - [ ] 飞书开放平台应用
  - [ ] 机器人 / Webhook
  - [ ] 文档、消息、任务等 API 权限
  - [ ] 本地 CLI 集成能力
- [ ] 规划技术架构：
  - [ ] 前端是否需要界面，还是主要依赖飞书卡片/CLI
  - [ ] 后端服务用什么栈
  - [ ] 知识库存储方式（轻量数据库 / 向量库 / 文件索引）
  - [ ] 任务调度与触发机制
- [ ] 梳理调用链：飞书数据获取 -> 处理 -> Agent 推理 -> 结果分发。
- [ ] 准备测试数据，避免后面因为没有真实场景而做不出 Demo。

## 七、Demo 前置设计
- [ ] Demo 不要贪大，先定义一个“最小可演示闭环”。
- [ ] 为 Demo 准备 3 类内容：
  - [ ] 原始输入样本
  - [ ] Agent 处理过程
  - [ ] 最终输出结果
- [ ] 提前设计演示脚本：
  - [ ] 用户做了什么
  - [ ] 系统感知到了什么
  - [ ] 提取了什么知识
  - [ ] 分发给了谁
  - [ ] 带来了什么收益
- [ ] 预留时间做录屏、截图、卡片展示和说明材料。

## 八、效果验证前置准备
赛题明确要求“证明价值”，这一部分必须提前设计。
- [ ] 设计评价维度：
  - [ ] 知识提取准确率
  - [ ] 行动项识别准确率
  - [ ] 推荐/分发相关性
  - [ ] 响应及时性
  - [ ] 用户满意度
  - [ ] 相比传统搜索/问答的效率提升
- [ ] 为每个维度准备简单可展示的验证方法：
  - [ ] 人工对照样本
  - [ ] 任务完成时间对比
  - [ ] 命中率统计
  - [ ] 用户主观打分
- [ ] 至少准备一组“使用前 vs 使用后”的对比数据。
