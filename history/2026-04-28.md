### 关键结论

#### 1. `schema` 的正确查询粒度需要先落到 `service.resource`

实测：

```bash
lark-cli schema im.messages
```

可返回该 resource 下的方法结构。

而此前自动生成的：

```bash

```

在当前环境里会报 `Unknown method`。

因此当前项目里与 schema 相关的知识规则，不应再默认建议：

```bash
lark-cli schema <service.resource.method>
```

而应先建议：

```bash
lark-cli schema <service.resource>
```

再根据 schema 返回内容定位具体 method 与参数结构。

#### 2. 这些 help 型建议命令当前都符合规范

经实测确认可用：

```bash

lark-cli sheets +write --help
lark-cli base +field-list --help
lark-cli wiki spaces get_node --help
lark-cli auth login --help
```

#### 3. 原始 IM 示例需要修正

此前样例写成：

```text
lark-cli im messages send --params '{"receive_id":"ou_xxx"}' failed: invalid params: missing requestBody.content
```

这个示例不够严谨，因为从当前 `im.messages` 的 schema 实测看：
- `receive_id_type` 属于 query / `--params`
- `receive_id` 属于 requestBody / `--data`

因此更合理的演示错误应当是：

```text
lark-cli im messages send --params '{"receive_id_type":"open_id"}' failed: invalid params: missing requestBody.receive_id
```

这样更能准确说明“把 body 字段漏掉了”的问题。

### 代码逻辑修正

文件：`src/app/runDemo.js`

#### 1. `maybeExecuteToolPlan` 现在会把命中的 `picked` 传入状态推断

原先：
- `inferPlannedStatus(toolPlan)` 只能看 `toolPlan.mode`
- `readonly_check` 一律被映射为 `command_ok_api_failed`

现在：
- `inferPlannedStatus(toolPlan, picked)` 可同时参考规则类别
- 对 `resource_access` 规则，`readonly_check` 会映射为 `command_ok_resource_not_found`
- 其他 `readonly_check` 仍保持 `command_ok_api_failed`

#### 2. 新增 `inferEffectiveStatus(actualStatus, plannedStatus)`

原先逻辑是：

```js
const effectiveStatus = toolExecution?.plannedStatus || actualStatus;
```

这会把真实执行结果覆盖掉，例如：
- `command_invalid`
- `command_ok_auth_failed`

都可能被错误覆盖为 `command_ok_help_returned`。

现在改为：
- 若真实执行结果已经是明确结论（如 `command_invalid`、`command_ok_auth_failed`、`command_ok_help_returned`、`command_ok_api_failed`、`command_ok_resource_not_found`、`permission_required`），则优先使用真实执行结果；
- 仅在 `blocked` / `not_executed` / 空值时，才退回到 `plannedStatus`。

这修复了此前 `docz` / `madeup` 一类 case 明明真实是 `command_invalid`，但回归时被误算成 `command_ok_help_returned` 的问题。
